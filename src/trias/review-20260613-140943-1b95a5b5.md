### 🔴 **HIGH SEVERITY**

#### **1. `config.py:48–52` — `_deep_merge` mutates its first argument in-place**
- **File**: `config.py`, lines 48–52  
- **Category**: *Correctness / Maintainability*  
- **Problem**: `_deep_merge(base, override)` modifies `base` directly. This is a **hidden side effect** — callers expect merging to be pure (especially since `load_config()` returns a deep copy of defaults). If `_deep_merge` were reused elsewhere (e.g., in tests or future modules), accidental mutation could cause subtle bugs.
- **Why it matters**: Violates *Locality* — changes propagate unpredictably. Also undermines *Interface* clarity: the function signature implies merge semantics, but behavior is destructive.
- **Fix**: Make `_deep_merge` return a new dict (or use `copy.deepcopy` + mutate copy).  
  ```python
  def _deep_merge(base: dict, override: dict) -> dict:
      merged = base.copy()
      for key, value in override.items():
          if key in merged and isinstance(merged[key], dict) and isinstance(value, dict):
              merged[key] = _deep_merge(merged[key], value)
          else:
              merged[key] = copy.deepcopy(value)  # avoid reference leakage
      return merged
  ```
  Then: `config = _deep_merge(config, user_config)`.

#### **2. `worker.py:106–113` — `check_model_health` incomplete / truncated**
- **File**: `worker.py`, lines 106–113 (truncated at `"num_pre`)  
- **Category**: *Correctness*  
- **Problem**: The function is cut off mid-payload construction. This likely breaks the health check entirely — missing closing braces, invalid JSON. Even if fixed, the payload uses `"prompt": "ping"` which is semantically incorrect: Ollama’s `/api/generate` expects a full prompt; `keep_alive: 0` may unload the model *during* the ping.
- **Why it matters**: A broken health check undermines reliability guarantees. Also violates *Locality*: error handling and retry logic are scattered (see line 123+), but core logic is incomplete → hard to debug/extend.

#### **3. `cli.py:57–68` — Remote file operations use shell-unsafe string interpolation**
- **File**: `cli.py`, lines 57–68  
- **Category**: *Security*  
- **Problem**: `_run(["ssh", REMOTE_HOST, "mkdir", "-p", str(uploads_dir)])` is safe (list args), but later:
  ```python
  _run(["ssh", REMOTE_HOST, "sh", "-c",
        f"ls {status_dir}/ 2>/dev/null || true"])
  ```
  uses f-string interpolation → **command injection** if `status_dir` contains shell metacharacters (e.g., `; rm -rf /`).  
- **Why it matters**: Even with `REMOTE_HOST`, untrusted config could inject malicious paths. Violates *Interface* safety — the seam assumes safe inputs, but `config["paths"]` is user-configurable.
- **Fix**: Avoid shell `-c`; use list args:
  ```python
  _run(["ssh", REMOTE_HOST, "ls", "-1", status_dir])
  ```
  Or better: use `pathlib.Path(status_dir).glob("*.json")` remotely via a *dedicated remote API* (see **Leverage** below).

### 🟠 **MEDIUM SEVERITY**

#### **4. `worker.py:32–51` — `FileLockAdapter` is shallow; no seam abstraction**
- **File**: `worker.py`, lines 32–51  
- **Category**: *Deep vs Shallow Module*  
- **Problem**: The adapter exposes platform-specific details (`fcntl`, `os.open`) and state (`_fd`). It’s a thin wrapper over OS primitives — low *Leverage*. Callers must know about file descriptors, locking modes, etc.  
  Worse: no interface (e.g., `Lock` protocol) is defined → testing requires mocking `fcntl.os`, not the adapter itself.
- **Fix**: Define an interface (e.g., `class LockSeam:` with `acquire() -> bool; release()`), then implement `FileLockAdapter(LockSeam)`. This makes the seam explicit, enabling:
  - Mock lock for tests
  - Fallback to no-op lock in non-POSIX environments  
  ```python
  class LockSeam(Protocol):
      def acquire(self) -> bool: ...
      def release(self): ...

  class FileLockAdapter(LockSeam): ...
  ```

#### **5. `config.py:72–80` — Path expansion is fragile and non-atomic**
- **File**: `config.py`, lines 72–80  
- **Category**: *Correctness / Maintainability*  
- **Problem**: Expanding paths in-place (`config["paths"]["mailbox"] = str(mailbox)`) mutates the config dict. If `load_config()` is called multiple times (e.g., for hot-reloading), paths may be double-expanded (`~/.trias` → `/home/user/.trias` → `/home/user/home/user/.trias`).  
- **Fix**: Store raw paths in config, expand at *usage time* (not load time). Or validate expansion is idempotent:
  ```python
  mailbox = Path(os.path.expanduser(config["paths"]["mailbox"]))
  if str(mailbox) != config["paths"]["mailbox"]:  # detect double-expansion
      raise ValueError("Path already expanded")
  ```

#### **6. `worker.py:128–140` — `ollama_generate` lacks input sanitization**
- **File**: `worker.py`, lines 128–140  
- **Category**: *Security / Correctness*  
- **Problem**: `model` and `prompt` are passed directly to Ollama. While `validate_model()` exists, it’s only used in `check_model_health()`. No validation on `prompt` → potential prompt injection or DoS via huge payloads.
- **Fix**: Enforce max prompt length (e.g., 100k chars) and validate model *in every call*, not just health checks.
