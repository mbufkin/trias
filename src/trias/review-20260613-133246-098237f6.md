2. **✅ Refactor locking to use a stateless seam + adapter pattern**  
   - *Rationale*: Current `acquire_lock` stores mutable state in function attribute (`_fd`), breaking thread/process isolation and violating locality (state scattered across calls). Replace with:  
     ```python
     class FileLockAdapter:
         def __init__(self, path): self._f = open(path, 'w+')
         def acquire(self): fcntl.flock(self._f.fileno(), LOCK_EX)
         def release(self): fcntl.flock(self._f.fileno(), LOCK_UN); self._f.close()
     ```  
   - *Leverage impact*: Enables mocking in tests (seam), supports multiple workers, and centralizes concurrency logic.

3. **✅ Make `DEFAULT_CONFIG` immutable/private**  
   - *Rationale*: Public mutation risks silent misconfiguration across the app. Wrap with `frozendict` or expose via read-only accessor.

   - *Interface benefit*: Makes the module’s seam (public API) more cohesive — users see all config-loading logic in one place.

### 🔴 HIGH SEVERITY

#### **1. Shell injection vulnerability in `worker.py:ollama_generate` (line ~80)**
- **File**: `worker.py`, line 79–85  
- **Category**: Security  
- **Description**: The `curl` command is built via string interpolation of `config['ollama']['url']` and `json.dumps(payload)` *inside* a shell command list. While `subprocess.run([...], ...)` avoids direct shell invocation, the payload JSON is passed as a raw string to `curl -d`. If any field in `payload` (e.g., `model`, `prompt`) contains untrusted input with shell metacharacters (e.g., backticks, `$(`), and if `shell=True` were accidentally added later—or if `curl` itself misparses the JSON—this becomes exploitable. More critically: **the prompt is user-controlled code**, and if it includes quotes or newlines, it could break JSON escaping *or* be interpreted by `curl` in unexpected ways (e.g., via `-d @-`).  
- **Fix**: Use `json.dumps(payload)` *and* pass it as a single argument to `curl`, but better yet: avoid shell entirely and use `requests` or `subprocess.run([...], input=json.dumps(payload).encode())`. Alternatively, ensure `curl` receives JSON via stdin with `-d @-` and pipe the payload.  
- **Leverage/Interface**: The interface (`ollama_generate`) should guarantee *no shell interaction*—currently it does not.

#### **2. Lock file race in `worker.py:acquire_lock` (line ~40–56)**
- **File**: `worker.py`, lines 41–56  
- **Category**: Correctness / Concurrency  
- **Description**: The lock is acquired *after* opening the file, but `fcntl.flock()` only works on open file descriptors. However, if two processes race to create the file (via `os.open(..., O_CREAT)`), and one creates it while the other opens an existing one, both may get different FDs—and both could succeed in locking *different* file descriptors pointing to the same inode. Worse: `fcntl.flock()` is **not** inherited across `fork()`, but the code assumes a single process context (no multiprocessing). Also, `acquire_lock._fd` is stored as a function attribute—this breaks if multiple workers run in threads or processes.  
- **Fix**: Use `with open(..., 'w+') as f: fcntl.flock(f.fileno(), ...)` and store the file handle *in the caller*, not as a global function attribute. Better yet: use `filelock` library or `fcntl.lockf()` on a dedicated lock file.  
- **Locality**: Locking logic is scattered across acquire/release with hidden state—violates locality.

#### **3. Model name validation too permissive in `worker.py:validate_model` (line ~90)**
- **File**: `worker.py`, line 87–91  
- **Category**: Security / Correctness  
- **Description**: Regex `r"[a-zA-Z0-9._:-]{1,128}"` allows colons (`:`), which are used in model names (e.g., `qwen3-coder-next:q4_K_M`)—but also allows *multiple* colons (e.g., `model:::evil`). Worse, it permits leading/trailing hyphens/dots (e.g., `-model`, `.model`), which may be interpreted as flags or paths by downstream tools. No check for null bytes, Unicode confusables (e.g., Cyrillic `о` vs Latin `o`), or control characters.  
- **Fix**: Enforce stricter format: e.g., `r"^[a-z0-9][a-z0-9._:-]*[a-z0-9]$"` and disallow consecutive separators. Or better: validate against a *whitelist of known models* (e.g., from `/api/tags`).  
- **Interface**: The interface should reject invalid model names *early*, not just warn.

### 🟠 MEDIUM SEVERITY

#### **4. `worker.py:check_model_health` is shallow and leaks implementation details**
- **File**: `worker.py`, lines ~93–120  
- **Category**: Deep vs Shallow Module, Interface  
- **Description**: This function does *too much*: validation + retry loop + jittered backoff + health ping. It’s a shallow module: complex interface (many parameters), little abstraction behind it. The retry/backoff logic should be in a *separate* retry utility—this is not the place for `time.sleep()` and exponential backoff. Also, health check uses same payload as generation (`"prompt": "ping"`), which may trigger model inference (wasteful).  
- **Fix**: Extract retry + jitter into `_retry_with_backoff()`. Health ping should use `/api/tags` or a lightweight endpoint.  
- **Leverage**: Low—caller must understand all retry semantics to use it.

#### **5. `config.py:_deep_merge` mutates `base` in-place (line ~70–78)**
- **File**: `config.py`, lines 69–78  
- **Category**: Correctness / Maintainability  
- **Description**: `_deep_merge` modifies the `DEFAULT_CONFIG` dict *in place* when merging user config. Since `load_config()` does `config = dict(DEFAULT_CONFIG)` (shallow copy), nested dicts like `"ollama"` are shared references—so mutations to `config["ollama"]` in one call affect future calls (e.g., if defaults are changed at runtime). This is a hidden side effect.  
- **Fix**: Use deep copy: `config = copy.deepcopy(DEFAULT_CONFIG)` or return a new dict (not mutate).  
- **Locality**: Bug would manifest across multiple config loads—hard to debug.

#### **6. Missing error handling in `cli.py:_submit_task` (line ~70–85)**
- **File**: `cli.py`, lines 72–91  
- **Category**: Correctness / Maintainability  
- **Description**: When uploading files, `_run(["scp", ...])` is called but only `result.returncode` is checked. If `scp` fails *before* sending (e.g., network unreachable), the error message (`stderr`) may be truncated or missing context. Also, no cleanup on partial failure: if file 1 uploads successfully and file 2 fails, the task is still submitted with incomplete files.  
- **Fix**: Check for `result.stderr` even if `returncode == 0`, and abort early on first upload failure. Log full error.  
- **Locality**: Error handling is ad-hoc—should be centralized.
